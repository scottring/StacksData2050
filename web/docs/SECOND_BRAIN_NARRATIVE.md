# The Second Brain: A Narrative

## The Idea

A solo founder running a 13-year-old SaaS company, managing Parkinson's disease, raising twins, searching for a backup job, and rebuilding his product from scratch has a finite amount of cognitive bandwidth. Every decision, every context-switch, every "where was I?" costs energy that doesn't regenerate on demand.

The Second Brain is the answer to a specific question: what if all of Scott's knowledge, priorities, relationships, and active projects lived in a structured, version-controlled, AI-readable system that could think alongside him?

Not a note-taking app. Not a task manager. A persistent external mind that any AI agent can read, query, and update — and that Scott can read directly when he wants to, because it's just markdown files in folders.

## The Architecture

### Layer 1: The Vault (Obsidian + Git)

The foundation is an Obsidian vault called `scotts-world`, stored at `~/Documents/scotts-world/` and version-controlled with Git. It's organized by type:

```
scotts-world/
  context/        — what's true right now
    scott-overview.md      — priorities, finances, decision framework
    stacks-data.md         — business overview
    stacks-clients.md      — Sappi, UPM, PPVIS, Omya, BASF
    stacks-revenue.md      — burn rate, runway, pricing model
    stacks-cdr.md          — CDR strategy (v1 built, v2 planned)
    job-search.md          — target roles, applications
    health.md              — Parkinson's, fitness, weight goals
    family.md              — Iris, Ella, Kaleb
    symphony-os.md         — secondary product vision
  projects/       — what's in flight
  tasks/          — what needs doing
  people/         — who matters
  outputs/        — what's been produced
  ideas/          — captured, not acted on
  inbox/          — raw captures, unprocessed
  daily/          — daily briefings and logs
```

Every file uses YAML frontmatter for metadata (type, status, domain, created, tags, linked). Files are cross-linked using Obsidian's `[[wiki-link]]` syntax. The vault auto-commits to Git every 5 minutes.

The key design choice: these are plain markdown files, not database rows. Scott can open them in Obsidian, read them in a text editor, grep them from a terminal, or let an AI agent parse them programmatically. The format is human-first, machine-readable by nature.

### Layer 2: Michael (The Chief of Staff)

Michael is a Node.js service running on Scott's MacBook. It uses the Claude Agent SDK to create AI agent sessions that read and write to the vault, and Grammy (a Telegram bot framework) to communicate with Scott via push notifications and conversational messages.

It has three scheduled tasks:

**Morning Briefing (5:00 AM)** — After CrossFit, Scott's phone buzzes with a tight summary: the top 1-2 things that move the needle today, overdue tasks, time-sensitive items, a health reminder. Under 15 lines. Generated by reading the vault's context files, tasks, and projects.

**Midday Check-in (12:00 PM)** — Three questions: Did you log food? What did you ship this morning? Any blockers? One message, no nagging.

**End of Day Capture (8:00 PM)** — Asks Scott what happened today. When he responds, Michael updates the relevant context files, marks completed tasks as done, creates new task files for anything that emerged, and gives a 3-line summary of what changed.

Michael also responds to ad-hoc messages and slash commands: `/new` decomposes a brain dump into tasks and projects; `/today` generates a daily briefing; `/capture` appends a raw thought to the inbox; `/update` modifies context files; `/review` runs a weekly review of all active items.

The key design choice: Michael's personality is defined entirely by its `CLAUDE.md` file, which it reads on every invocation. No em dashes. No sycophancy. No narrating what it's about to do. Verb-first action items. Summary first, details on request. It exists to help Scott focus and execute, not to add complexity.

### Layer 3: Specialist Agents (The Departments)

This layer is designed but not yet built. The concept: Michael acts as a chief of staff who can delegate to specialist agents for different business functions. Each specialist gets the right slice of vault context and a role-specific system prompt.

The departments:

**Engineering** — Spawns Claude Code sessions against specific project directories (`~/Developer/StacksData2050/`, `~/Documents/compliance-data-router/`). This is the only department that needs code execution capabilities; all others run as Agent SDK calls against the vault.

**Business Development** — Reads `stacks-clients.md`, `stacks-cdr.md`, relevant people files. Prepares call briefs, drafts outreach, tracks follow-ups. When Scott says "prep me for the Zahra call," this agent reads everything about BASF and the CDR and produces a structured brief.

**Finance** — Reads `stacks-revenue.md`. Tracks burn rate, flags when runway assumptions change, reminds about invoicing.

**Job Search** — Reads `job-search.md`. Tracks applications, tailors resume bullets for specific roles, preps for interviews.

**Health** — Reads `health.md`. Tracks weight, food logging compliance, medication schedule, exercise modifications for injuries.

Each "department" is not a separate application. It's a different system prompt and context selection. Michael routes based on the task domain, constructs the specialist's context window from the relevant vault files, and collects results back into the vault.

## The Open Brain Connection

The architecture is inspired by Nate B Jones's "Open Brain" concept: a personal knowledge system where your thoughts are stored as vector embeddings in a Postgres database, and any AI agent can query them semantically via MCP (Model Context Protocol). You think something once, it gets embedded, and you never have to re-explain yourself.

The Second Brain follows the same structural pattern:

```
Open Brain:      thoughts → embeddings → any AI queries → never re-explain
Second Brain:    vault files → structured context → any agent reads → never re-brief
```

The vault is the canonical store. Michael is the nervous system. Specialist agents are the limbs. Scott is the executive function — he sets direction, makes decisions, and reviews results. Everything else is delegated.

### Where Vector Embeddings Fit

Today, the vault is searched via Obsidian's built-in search (text matching) and Michael's file reads (direct access). This works because the vault is small — dozens of files, not thousands.

If the vault grows to hundreds of files, or if Scott wants semantic search (find related concepts, not just keyword matches), vector embeddings become valuable. The path:

1. Each vault file gets embedded via OpenAI's text-embedding-3-small (the same model planned for the CDR)
2. Embeddings stored in a Postgres + pgvector database (local or Supabase)
3. Michael queries semantically: "What do I know about BfR compliance?" returns relevant vault files ranked by semantic similarity, not just files containing the word "BfR"
4. Embeddings regenerate on file change (Git commit hook triggers re-embedding of modified files)

This is the same pattern as the CDR v2, applied to personal knowledge instead of compliance data. The vault ages like wine too: every file added, every context update, every captured thought makes the semantic index richer and more queryable.

The infrastructure is identical: pgvector, HNSW indexes, cosine similarity. The embedding model is the same. The query pattern is the same. If Scott builds it for one system, he has it for both.

### MCP as the Connectivity Layer

The Open Brain uses MCP (Model Context Protocol) to expose its semantic search as tools that any AI agent can call. The Second Brain can do the same: an MCP server that exposes `search_vault(query)`, `read_context(domain)`, `update_context(domain, content)`, and `create_task(title, details)`.

Any AI agent — Michael, Claude Code, a future web-based assistant — could connect to the vault's MCP server and have full read/write access to Scott's structured knowledge. The vault becomes a universal backend for personal AI.

## What Could Go Wrong

### Over-engineering the personal

The vault works today as plain markdown files. Adding vector embeddings, MCP servers, and semantic search is technically clean but practically unnecessary until the vault reaches a scale where text search fails. If Scott spends a week building personal infrastructure instead of demoing the CDR to Abde, that's a week of runway burned on the wrong problem.

The mitigation: the vault reorg (splitting `stacks-data.md` into departmental files) is the only change needed now. Vector embeddings and MCP are future layers that add value at scale, not at the current size.

### Context window as bottleneck

Michael reads vault files into its context window on every invocation. The Agent SDK has limits. If the vault grows to hundreds of files and Michael tries to read all relevant context for a complex task, it hits the window ceiling. This is the same problem the Open Brain solves with embeddings: instead of reading everything, query semantically and read only what's relevant.

For now, the vault is small enough that selective file reads work. The context file table in Michael's `CLAUDE.md` tells it which files to read for which topics. As long as that table is maintained, the bottleneck doesn't bite.

### Single-device dependency

Michael runs on Scott's MacBook via launchd. Laptop closed means no agent. The Week 3 plan moves it to a Mac Mini for 24/7 uptime, but until then, scheduled tasks only fire when the laptop is open and awake.

The mitigation is acceptable: morning briefings fire after CrossFit when Scott opens his laptop. Midday and evening tasks fire if the laptop is open, which it usually is during work hours. The Mac Mini is the real fix.

### Trust and autonomy boundaries

Week 1 Michael only asks questions and reads the vault. It doesn't autonomously write to files or execute code. Week 2 adds vault write-back. Week 3 adds autonomous overnight work. Each step increases both capability and risk.

The vault is Git-versioned, so any bad write can be reverted. But an agent that autonomously marks tasks as done, creates new tasks, or updates context files could introduce errors that propagate through future briefings. The trust boundary needs to expand slowly, with Scott reviewing changes before they become the new baseline.

## Chances of Working

### What works in favor

**The vault already exists and is useful.** Scott uses it daily. The reorg into department-aligned files makes it more machine-readable without breaking the human experience. This isn't a system that needs to be built from scratch — it needs to be wired up.

**Michael is mostly built.** The bot, the agent SDK integration, the scheduler, the memory system, the process management — all working. The gap is configuration (CLAUDE.md rewrite, scheduled task seeding, .env setup) and deployment (launchd plist). Estimated: 75 minutes of setup.

**The department model is just prompt engineering.** Specialist agents aren't separate applications. They're different system prompts with different context selections. Adding a new "department" is writing a prompt template and mapping it to vault files. No infrastructure needed.

**Obsidian + Git is rock-solid.** The foundation — markdown files in a Git repo — is the simplest possible architecture. It will never break in a way that loses data. It will never require a migration. It will never have a vendor lock-in problem. Every future layer (embeddings, MCP, semantic search) is additive and optional.

### What works against

**Maintenance burden.** A personal knowledge system is only as good as its maintenance. If Scott stops updating context files, the morning briefings become stale and eventually useless. The system requires consistent input to produce useful output. This is a discipline problem, not a technical one, but it's real.

**Diminishing returns on complexity.** The vault with Michael is useful. Adding vector embeddings makes it marginally more useful. Adding MCP makes it marginally more useful again. Each layer has diminishing returns relative to its complexity. The risk is spending engineering time on personal infrastructure optimization instead of on the product that generates revenue.

**AI Agent SDK stability.** Michael depends on the Claude Agent SDK, which is evolving rapidly. API changes, session management updates, and tool availability shifts could require maintenance work that Scott doesn't have bandwidth for during crunch time.

### The honest assessment

The Second Brain is working infrastructure, not speculative architecture. The vault exists. Michael exists. The department model is a natural extension. The vector embedding layer is future-proofing that pays off if and when the vault grows large enough to need it.

The smart play: finish Michael Week 1 setup (75 minutes). Let it run for a week. See if the morning briefings actually change Scott's behavior and focus. If they do, proceed to Week 2 (vault write-back, work sessions). If they don't, the vault still works as a static knowledge base.

The Second Brain doesn't need to be perfect. It needs to do one thing well: make sure Scott wakes up every morning knowing exactly what to work on, and goes to bed every night having captured what happened. Everything else — specialist agents, vector search, MCP connectivity — is gravy.
